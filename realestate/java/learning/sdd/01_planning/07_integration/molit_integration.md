# 외부 연계 설계 (07_integration): data.go.kr 국토교통부 실거래가

> 2단계 산출물입니다. 실재 공개 API와의 연계 계약, 인증키 관리, 페이징, 회복력, rate limit 대응, 스케줄링, 장애 격리, 정합 검증을 정본으로 박제합니다.
> 외부 명세 정본: `00_sources/01_apis/molit_apt_trade_api.md`. 추적: SIR-001·002·005, SECR-001~003, AC-1·2, PER-002·004.

## 2. 요청 파라미터 (SIR-001)

| 파라미터 | 의미 | 비고 |
| --- | --- | --- |
| serviceKey | 인증키 | 환경변수 `MOLIT_SERVICE_KEY`로만 주입(SECR-001). Decoding 키 사용 |
| LAWD_CD | 법정동코드 5자리(시군구) | 예: 서울 종로구 `11110` |
| DEAL_YMD | 계약월 YYYYMM | 예: `202405` |
| pageNo | 페이지번호 | 1부터 |
| numOfRows | 페이지당 건수 | 최대 1000으로 호출 수 최소화(PER-002) |

---

## 3. 응답 처리 (SIR-002)

- 응답은 XML입니다. 수집 서비스가 XML을 파싱해 표준 스키마(AptTransaction)로 변환합니다.
- 헤더 `resultCode`가 `000`이 아니면 본문을 적재하지 않고 결과코드별 정책(아래 5절)으로 분기합니다.
- 매핑·변환 규칙은 `04_data/realprice_data.md`를 정본으로 합니다(금액 만원→원, 문자열→숫자, 해제 코드→boolean, `YY.MM.DD`→date).

수집 대상 핵심 item 필드:
`sggCd`, `umdNm`, `jibun`, `aptNm`, `aptSeq`, `excluUseAr`, `dealYear/Month/Day`, `dealAmount`(만원·콤마), `floor`, `buildYear`, `dealingGbn`, `estateAgentSggNm`, `cdealType`(해제여부 `O`), `cdealDay`(해제사유발생일), `rgstDate`(등기일자), `slerGbn`/`buyerGbn`(매도/매수자), `aptDong`(동, 등기 완료 건만), `landLeaseholdGbn`(토지임대부 여부).

## 4. 인증키 관리 (SECR-001~003)

| 항목 | 정책 |
| --- | --- |
| 저장 | 코드·이미지·Config 파일에 평문 금지. 환경변수 `MOLIT_SERVICE_KEY`만 사용 |
| 외부화 | Config Server는 엔드포인트·회복력 정책만 관리하고, 인증키는 환경변수 참조로 주입(SECR-002) |
| 로그 | 요청 URL·로그·트레이스·에러 응답에서 인증키 마스킹(SECR-003) |
| 교체 | 만료·교체 시 환경변수 갱신 + 설정 리프레시로 무중단 반영(SECR-005) |
| 형태 | Decoding 키 사용. URL 조립 시 이중 인코딩 주의 |

## 5. 결과코드별 처리 (회복력 분기)

| resultCode | 의미 | 처리 |
| --- | --- | --- |
| 000 | 정상 | item 정규화·적재 |
| 03 | NODATA | 정상 종료(빈 결과). 해당 (시군구·월) 0건으로 간주 |
| 01 / 04 / 99 | 일시 오류 | 재시도 대상(아래 6절) |
| 22 | 트래픽 초과 | 백오프 후 재시도 또는 익일 재개(7절) |
| 12 / 20 / 30 / 31 / 32 | 키·권한·서비스 문제 | 즉시 실패·운영 알림. 재시도하지 않음 |

---

## 6. 회복력 정책 (AC-2 / SIR-005)

resilience4j 기반. 외부 장애를 수집 경계 안에 가둡니다.

| 정책 | 값 | 근거 |
| --- | --- | --- |
| 타임아웃 | connect 3s / read 10s | 외부 지연이 수집 스레드를 무한 점유하지 않게 |
| 재시도 | 3회, 0.5s 간격(지수 백오프 옵션) | 일시 오류(01/04/99) 흡수 |
| 서킷브레이커 | sliding window 20, 실패율 50%, open 10s, half-open 시도 3 | 연속 실패 시 빠른 차단·자동 회복 |
| fallback | 빈 결과 반환(부분 수집 허용) | 한 구간 실패가 배치 전체를 멈추지 않게(SFR-011) |
| bulkhead(선택) | 동시 외부호출 제한 | 트래픽 한도·자원 보호 |

> 회복력은 수집 서비스(ingestion)에만 적용합니다. 거래원장·분석은 외부를 직접 호출하지 않으므로 외부 장애의 영향권 밖입니다(아키텍처 판단 2, 장애 격리).

## 7. Rate limit 대응 (CONR-001 / PER-004)

| 항목 | 정책 |
| --- | --- |
| 일일 한도 | 개발계정 10,000건 / 운영계정 증액 신청 |
| 호출 절감 | numOfRows=1000으로 페이징 호출 수 최소화(PER-002) |
| 동시성 제어 | 다중 시군구 수집의 동시 실행 수를 제한해 한도 초과 방지 |
| 22 코드 대응 | 트래픽 초과(22) 수신 시 백오프, 한도 소진이면 익일 스케줄로 이월 |
| 호출 회계 | 일일 호출 수를 카운트해 한도 임계 도달 시 수집 속도 조절 |

## 8. 페이징 수집 (SFR-002)

```
1) pageNo=1, numOfRows=1000 호출
2) 응답 totalCount 확인
3) 누적 수신 < totalCount 이면 pageNo++ 반복
4) 마지막 페이지까지 수집(누락 0 보장)
5) 페이지 단위 실패는 재시도(6절). 끝내 실패한 페이지는 skipped로 기록(부분 수집)
```

## 9. 스케줄링 (SFR-010)

| 트리거 | 설명 |
| --- | --- |
| 정기 배치 | 매월 (250개 시군구 × 직전 계약월) 조합 수집 |
| 백필 | 과거 계약월 구간 재수집(멱등, SFR-013). 신고 지연·해제 반영 목적 |
| 수동 트리거 | `POST /api/v1/ingest/apt-trade` 단건 수집 |

- 신고·해제는 사후 변동되므로(CONR-005), 최근 수개월은 재수집 대상에 포함해 변동을 따라잡습니다.

---

## 10. 장애 격리 (아키텍처 판단 2)

- 수집을 별도 서비스(ingestion-service)로 분리해 외부 API 장애·트래픽 제한이 거래원장 쓰기·분석 조회로 전파되지 않게 합니다.
- 수집 실패 시: 성공 구간은 멱등 적재, 실패 구간은 skipped 기록 후 다음 배치/백필에서 재시도. 조회·통계는 영향받지 않습니다.

---

## 11. 데이터 정합 검증

| 검증 | 내용 |
| --- | --- |
| 건수 정합 | 페이징 누적 수신 == totalCount 확인. 불일치 시 해당 구간 재수집 |
| 금액 정합 | dealAmount 파싱 실패 건은 적재 거부·스킵·보고(DAR-002) |
| 멱등 정합 | 동일 구간 2회 수집 시 거래원장 행 수 불변(자연키 유니크, AC-4) |
| 해제 정합 | cdealType=`O` 건은 canceled=true로 적재되고 집계에서 제외됨(AC-3). 사후 해제 전환 반영 확인 |
| 결측 허용 | 동(aptDong)·등기일자(rgstDate)는 등기 완료 전 공백 허용(CONR-004) |

## 12. 부속 원천: 법정동코드

- LAWD_CD(시군구 5자리)는 행정표준코드관리시스템 기준입니다.
- 조회: `https://www.code.go.kr/stdcode/regCodeL.do` (법정동코드 10자리 중 앞 5자리).
- 250개 시군구 코드를 마스터로 보관하고 매월 수집을 구동합니다(DAR-008).
